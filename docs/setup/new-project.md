# Новый проект — как подключить `ai-settings`

## Короткий путь (если проекту хватает глобальных правил)

Ничего не делать. Глобальные правила применяются автоматически через:
- `~/.claude/CLAUDE.md` (Claude Code),
- `~/.codex/AGENTS.md` (Codex CLI),
- `~/.gemini/GEMINI.md` (Gemini CLI).

Для Cursor прогони один раз в проекте:

```bash
~/ai-settings/scripts/sync-cursor.sh --project .
```

## Длинный путь (проектные специфики)

В корне проекта:

```bash
~/ai-settings/scripts/init-project.sh
```

Создаёт:
- **`AGENTS.md`** (project-local, additive) — заполни TODO-блоки по факту проекта.
- **`.claude/settings.json`** — пустой override. Можно добавить разрешения для команд, специфичных проекту (например, `docker-compose up`).
- **`CHANGELOG.md`, `TODO.md`** — шаблоны, если их не было.
- **`.cursor/rules/ai-settings.mdc`** — плоская копия глобальных правил для Cursor.
- **`.gitignore`** — добавляет AI-артефакты (`.claude/sessions/`, `.claude/cache/`, `.cursor/sessions/`).

## Типичные проектные правки в `AGENTS.md`

- Какие фреймворки и их версии используются.
- Ссылки на проектные ADR и архитектурные решения.
- Конвенции команды, отличающиеся от глобальных (например, другой стайлгайд тестов).
- Команды сборки / запуска / тестирования, специфичные для проекта.

## Чего НЕ стоит класть в проектный `AGENTS.md`

- Дублировать глобальные правила — они уже применяются.
- Secrets или приватные конфиги — это не про `AGENTS.md`.
- Детали, которым место в `README.md` для людей (инструкции по запуску, описание архитектуры).

## После init

Если CHANGELOG.md уже велся — скрипт его не трогает. Если нет, созданный шаблон Keep a Changelog можно сразу использовать — смотри скилл `changelog-entry` в `~/ai-settings/skills/code/changelog-entry/`.
